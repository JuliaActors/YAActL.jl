Actors

An Actor is a computational entity that, in response to a message it receives, can concurrently:

  • send a finite number of messages to other actors;
  • create a finite number of new actors;
  • designate the behavior to be used for the next message it receives. [1]

YAActL actors are Julia tasks represented by local or remote LINKs, channels over which they receive and send messages [2]. They:

  • react to those messages,
  • execute user defined behavior functions when receiving certain messages,
  • change their behavior upon request,
  • update their internal state influencing how they behave.

YAActL provides various commands to set, control, trigger and query actor behavior and state.

Start

In the simplest case we use Actor to start an actor with a previously implemented behavior and get a link to it.

The following code starts an actor with a stack_node behavior function and some arguments to it.

mystack = Actor(stack_node, StackNode(nothing, Link()))

We can now send! messages to mystack or do other operations on it.

Messages

YAActL actors operate on predefined messages all of type Message. Basically there are only two functions to interact with an actor:

  • send!: send a message to an actor,
  • receive!: receive a message,

Actors process messages asynchronously.

A user can implement further message types. For example:

struct Pop <: Message
    customer::Link
end

struct Push{T} <: Message
    content::T
end

Those are forwarded by the actor as last arguments to its behavior function.

Behavior

Actors execute their behavior function when they receive a Request message or another user implemented message [3].

They pass those messages as the last argument to the behavior function. How actors compose the arguments, is explained in length in Behaviors.

The actor will store the return value in its internal res variable. It can be queried from the actor with query!.

Actor Control

Actors can be controlled with the following functions:

  • become!: cause an actor to switch its behavior,
  • cast!: cause an actor to execute its behavior function,
  • exit!: cause an actor to terminate,
  • init!: tell an actor to execute a function at startup,
  • set!: set an actor's dispatch mode,
  • term!: tell an actor to execute a function when it terminates.
  • update!: update an actor's internal state,

Those functions are wrappers to predefined messages and to send!.

Actors can also operate on themselves or rather they send messages to themselves:

  • become: and actor switches its own behavior,
  • self: an actor gets a link to itself,
  • stop: an actor stops.

Bidirectional Messages

What if you want to receive a reply from an actor? On top of the asynchronous messaging between actors there is an interface with bidirectional synchronous messages:

  • request!: send a message to an actor and receive the result.

The following functions use request! and do specific things with it:

  • call!: call an actor to execute its behavior function and to return the result,
  • get!: get an actor's internal state,
  • exec!: tell an actor to execute a function,
  • query!: prompt for the result of the last call to the behavior function.

All those functions - if you don't provide a response link - will establish a private link to an actor, block, receive the result and return it. You should not use blocking when you need to be strictly responsive.

Actor State

The actor state is internal and is shared with its environment only for diagnostic purposes. The API functions above are a safe way to access actor state.
