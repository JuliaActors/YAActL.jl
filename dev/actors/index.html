Actors

An Actor is a computational entity that, in response to a message it receives, can concurrently:

  • send a finite number of messages to other actors;
  • create a finite number of new actors;
  • designate the behavior to be used for the next message it receives. [1]

YAActL actors are Julia tasks represented by local or remote LINKs, channels over which they receive and send messages [2]. They:

  • react to those messages,
  • execute user defined behavior functions when receiving certain messages,
  • change their behavior upon request,
  • update their internal state influencing how they behave.

YAActL provides various commands to set, control, trigger and query actor behavior and state.

Start

In the simplest case we start an Actor with a previously implemented behavior and get a link to it.

YAActL.Actor — Function
Actor([lp::LinkParams], bhv::Function, args...; kwargs...)
Actor(pid::Int, bhv::Function, args...; kwargs...)

Create a new actor. Start a task executing repeatedly the behavior bhv. The actor listens to messages msg sent over the returned link and executes bhv(args..., msg, kwargs) for each message. The actor stops if sent Stop().

Arguments

  • [lp::LinkParams]: optional parameters for creating the actor,
  • pid::Int: create the actor on process pid,
  • bhv::Function: function implementing the actor's behavior,
  • args...: arguments to bhv, (without msg)
  • kwargs...: keyword arguments to bhv.

Returns

  • a Link to a locally created actor or
  • an RLink to a remote actor.
source

The following code starts an actor with a stack_node behavior function and some arguments to it.

mystack = Actor(stack_node, StackNode(nothing, Link()))

We can now send! messages to mystack or do other operations on it.

Operation

An actor recognizes and operates on predefined messages. Basically there are only two functions to interact with an actor:

  • send!: send a message to an actor,
  • receive!: receive a message,
  • request!: send a message to an actor, block, receive and return the result.

source

If an actor receives a Request message or any other not predefined message, it executes its behavior function with this message as the last argument.

This mechanism can be employed to extend the actor's behavior. A user can implement further message types. For example:

struct Pop <: Message
    customer::Link
end

struct Push{T} <: Message
    content::T
end

User API

The following interface to actors is for common use:

  • become!: cause an actor to switch its behavior,
  • call!: call an actor to execute its behavior function and to return the result,
  • cast!: cause an actor to execute its behavior function,
  • get: get an actor's internal state,
  • init!: tell an actor to execute a function at startup,
  • query: prompt for the result of the last call to the behavior function,
  • self: get your actor's link,
  • set!: set the actor's dispatch mode,
  • stopActor!: terminate an actor,
  • terminate!: tell an actor to execute a function when it terminates.
  • update!: update an actor's internal state,

Those functions are wrappers to the predefined messages and to the send! or request! functions. They all involve a communication.

Actors can also operate on themselves or rather they send messages to themselves:

  • become: and actor switches its own behavior,
  • self: an actor gets a link to itself,
  • stopActor: an actor stops.

Actor State

The actor state is internal and is shared with its environment only for diagnostic purposes. The API functions above are a safe way to access actor state.
